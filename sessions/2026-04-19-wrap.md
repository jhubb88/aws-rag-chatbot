### Graceful Rate-Limit Handling (post-swap)

**Problem found during recruiter smoke test:** SambaNova returned HTTP 429 Rate Limit Exceeded when hit with burst queries during the heavy session. Lambda was returning generic HTTP 500; frontend showed "API returned HTTP 500 ()." — terrible UX for a demo.

**Root cause:** SambaNova free tier enforces 240 RPM. Today's session burned through the burst limit (session validation queries + warmup pings + recruiter testing in quick succession).

**Fix:**
- Lambda: detects 429, logs `retry-after` / `x-ratelimit-remaining` / `x-ratelimit-reset` headers (all returned `unknown` — SambaNova doesn't send these headers), returns HTTP 200 with `{"error_type": "rate_limit", "message": "...", "engine_used": null}`.
- Frontend: detects `error_type === "rate_limit"` and renders a "Rate Limited" card with "Switch to AWS Bedrock" guidance instead of a red error card.

**Sentinel pattern:** `_generate_sambanova` returns `(None, "sambanova_rate_limited")` on 429 — caught in `lambda_handler` before the generic failure check. Inline comments at both sites: "keep string in sync."

**Verified live:** Rate-limit card rendered correctly in Playwright screenshot on `https://d1r1qv7io7k8vk.cloudfront.net`.
