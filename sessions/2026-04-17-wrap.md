### Performance Fix (pre-Step 7 blocker)
WA queries were hitting the API Gateway 29s hard timeout. Three fixes applied:

1. **QueryLambda MemorySize 256MB → 1024MB** — CloudFormation template updated + direct Lambda update (CFN deploy had an unrelated `ResourceExistenceCheck` validation error; template is correct for future full deploys)
2. **Module-level index caching** — `_index_cache` global in `src/lambdas/query/handler.py`; confirmed via CloudWatch "Using cached vector index" log on warm invocations
3. **max_tokens 512 → 256** — both Nebius (line 235) and Bedrock (line 311) paths matched

Cold worst-case: 32,279ms → 14,860ms (~54% reduction). Warm queries: 5–8s. All queries pass under 29s.

### Permanent Observability Logging (do NOT remove)
Two log lines in `_generate_nebius` in `src/lambdas/query/handler.py`:
- `[DEBUG] Nebius request body: {model, max_tokens, temperature}` — confirms serialized payload before send
- `[INFO] Nebius usage: prompt_tokens=N completion_tokens=N finish_reason=X` — ground truth from Nebius after response

These resolved an ambiguous diagnosis: Nebius was suspected of ignoring `max_tokens`. The usage log confirmed `completion_tokens=230, finish_reason=stop` — parameter honored, model answered naturally within budget.

**Diagnostic lesson recorded:** Llama 3.3-70B tokenizes technical AWS content at ~5 chars/token. 256 tokens = ~1,150–1,280 chars on cost/WA content. Char count alone is unreliable as a proxy for token count on domain-specific text.
