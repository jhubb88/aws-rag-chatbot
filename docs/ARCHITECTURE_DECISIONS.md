# ARCHITECTURE DECISIONS — RAG Knowledge Chatbot

## How the RAG Pipeline Works — Step by Step

### Ingestion (one-time setup)

1. Jimmy drops source documents into `data/documents/` (PDF, TXT, or MD files)
2. An ingestion Lambda reads each file from S3
3. The Lambda splits each document into chunks (roughly 500 tokens each, with overlap so context is not lost at chunk boundaries)
4. Each chunk is sent to Amazon Titan Embeddings v2, which returns a 1536-dimension vector representing the chunk's meaning
5. Each chunk and its vector are saved as a JSON object in an S3 "vector store" bucket
6. When all documents are processed, the knowledge base is ready

### Query (every user request)

1. User types a question in the browser
2. Request hits API Gateway (HTTPS POST)
3. API Gateway triggers the query Lambda
4. Query Lambda sends the question to Titan Embeddings v2 → gets a question vector
5. Lambda loads all stored chunk vectors from S3
6. Lambda computes cosine similarity between the question vector and every chunk vector
7. Top N chunks (by similarity score) are selected
8. Lambda builds a prompt: system instructions + retrieved chunks + user question
9. Prompt is sent to Claude 3 Haiku via Bedrock
10. Claude returns an answer grounded in the retrieved chunks
11. Answer is returned through API Gateway to the browser

## Custom Retrieval Layer

**Why we are NOT using Bedrock Knowledge Bases:**

Bedrock Knowledge Bases is a managed service that does the embedding, storage, and retrieval for you automatically. It is convenient but it hides all the engineering. A recruiter looking at this project would see one AWS service call, not a retrieval system.

**What we are doing instead:**

- Embeddings are generated by calling Titan Embeddings v2 directly from Lambda
- Vectors are stored as JSON files in S3 (one file per chunk, or a single manifest file)
- Retrieval uses cosine similarity computed in the Lambda function
- This means the query Lambda loads vectors, scores them, selects the top matches, and builds the prompt — all visible, all testable, all explainable

**Cosine similarity in plain English:**

Two vectors point in the same direction if they have similar meaning. Cosine similarity measures the angle between them — a score of 1.0 means identical direction (same meaning), 0.0 means unrelated. The Lambda picks the chunks with the highest scores.

## Why CloudFormation Is in MVP

Infrastructure that is not code is infrastructure that cannot be reproduced. If the S3 bucket, Lambda, and API Gateway are created by clicking around in the AWS console, they cannot be version-controlled, reviewed, or recreated after deletion. CloudFormation makes the entire infrastructure declarative and repeatable. It also demonstrates a key skill expected of a systems engineer working with AWS.

---

## Why Cognito and CloudFront Are Phase 2

Both services add significant complexity and cost that is not justified for a portfolio demo:

- **Cognito** requires user pool setup, JWT handling in the frontend, and Lambda authorizer configuration. The demo does not need accounts.
- **CloudFront** requires a custom domain (Route 53 or external registrar), ACM certificate provisioning, and cache behavior configuration. The S3 static site is sufficient for demo access.

Both will be added in Phase 2 if the project grows into a real product.

## Why Claude 3 Haiku

| Model | Speed | Cost | Quality for Q&A |
|-------|-------|------|----------------|
| Claude 3 Haiku | Fast (< 2s typical) | $0.25/1M input tokens | Excellent for factual Q&A on retrieved context |
| Claude 3 Sonnet | Medium | ~4x more expensive | Better for complex reasoning — overkill here |
| Claude 3 Opus | Slow | ~15x more expensive | Best overall — not justified for this use case |

Haiku is the right choice for a RAG chatbot where the answer is constrained by retrieved context. The model does not need to reason deeply — it needs to summarize and synthesize retrieved text accurately and quickly.
